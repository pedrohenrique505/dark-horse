Para um site de xadrez funcional, eu definiria esta stack:

Stack recomendada

Frontend

Next.js + React + TypeScript
Chessground para o tabuleiro
chess.js para regras, validação de jogadas, xeque, mate, empate e FEN/PGN

Tempo real / multiplayer

Node.js no backend
Socket.IO para partidas online em tempo real

Banco

PostgreSQL
Prisma ORM

Infra

Redis para presença, matchmaking, fila e rate limiting
Docker para subir tudo de forma previsível
Por que essa stack
1) Tabuleiro: use Chessground

Para o que você pediu, essa é a escolha mais direta. O Chessground já traz:

desenho de setas e círculos em SVG
setas que “snapam” para lances válidos
suporte a premove por clique ou drag
callbacks e reconfiguração em tempo real
layout fluido e suporte mobile

Ou seja: pre-moves e desenhar setas com o botão direito já estão no perfil da biblioteca, em vez de você ter que inventar isso por cima.

2) Regras do jogo: use chess.js

O Chessground é UI; ele mesmo diz que não tem lógica de xadrez embutida. Para regras, o complemento natural é o chess.js, que faz geração e validação de lances, além de detectar xeque, mate, afogamento e empates.

3) Multiplayer: use Socket.IO

Para jogo online você precisa comunicação de baixa latência, bidirecional e orientada a eventos entre cliente e servidor. O Socket.IO oferece isso e ainda inclui fallback de transporte e reconexão automática, o que ajuda bastante em partidas reais. Rooms também servem muito bem para modelar “uma sala = uma partida”.

4) App web: Next.js

Para site completo, com landing page, login, perfil, histórico, ranking e páginas dinâmicas, eu iria de Next.js com App Router. Ele já estrutura bem a aplicação React moderna e ajuda a manter frontend e partes server-side no mesmo projeto.

5) Persistência: PostgreSQL + Prisma

PostgreSQL é uma escolha segura para usuários, partidas, ratings, histórico, relações entre jogadores, convites e torneios. O Prisma acelera muito o desenvolvimento com tipagem e migrations.

Ponto importante sobre licença

Aqui está o único cuidado grande: Chessground usa GPL-3.0 e o repositório afirma que, ao usar Chessground no seu site, o trabalho combinado pode ser distribuído apenas sob GPL e que você deve liberar o código-fonte aos usuários do site.

Então:

se o projeto for open source, eu seguiria com Chessground
se o projeto for fechado/comercial proprietário, eu não escolheria Chessground sem antes validar isso juridicamente
Se você quiser projeto fechado/comercial

A alternativa mais segura é trocar o board por um componente React mais permissivo, como react-chessboard, que é moderno e responsivo, mas ele não declara nesses trechos a mesma profundidade de suporte nativo a premove/setas que o Chessground mostra explicitamente. Então, nesse cenário, você provavelmente implementaria parte dessas interações por cima.

Minha decisão objetiva
Eu escolheria isto:

Next.js + TypeScript + Chessground + chess.js + Socket.IO + PostgreSQL + Prisma + Redis

Só mudaria uma coisa:

Se o site for proprietário, eu trocaria Chessground por outra board com licença mais conveniente e aceitaria implementar manualmente parte de:
